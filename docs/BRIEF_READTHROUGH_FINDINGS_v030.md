# v0.30.0 prep — real-case brief read-through findings

Generated by running `python scripts/smoke_deliverables.py` against the
bundled ALEC-TEST-2026 fixture (1,067 transfers, 7 USDC recovery
targets, 4 FREEZABLE / 3 INVESTIGATE) and reading each artifact as a
detective / federal agent / law firm recipient would.

## Tier-1 (CRITICAL — would discredit the brief to a real recipient)

### F1. Version footer pinned to v0.16.4 (live brief says we're 13 releases old)

`templates/le.html.j2` and friends render `Generated ... · v0.16.4` in the
footer. We're at v0.29.1. Any LE recipient eyeballing a forensic package
with a 13-version-stale version stamp would question whether the
operator is paying attention. Fix: pull from `recupero.__version__` or
the brief manifest, not a hardcoded string.

### F2. Victim PII leaks to issuer compliance team

The freeze request to Circle (`freeze_request_circle_*.html`, line 865)
includes the victim's:
- Full home address: `4925 Riverbend Dr, Fort Worth, TX 76102`
- Email: `alec@recupero.io`

Circle's compliance team has no need for this — they need wallet
addresses, token contracts, perpetrator wallets, and a *requester
point-of-contact* (which should be Recupero LLC / the law firm, not the
victim). The LE handoff package is the appropriate vessel for victim
PII; the issuer freeze letter is not.

### F3. Jurisdiction misdetection — US victim flagged as non-US

Cover meta and victim summary correctly show "USA (Texas)". But Section
6.1 "Suggested Filing Routes" emits:
- Channel: "National cybercrime / financial-crime authority"
- Jurisdiction: "Non-US (consult attorney)"
- Filing notes: **"Victim located outside the US (USA (Texas))"**

The jurisdiction-detection logic is treating "USA (Texas)" as non-US.
Expected behavior: US victim → IC3 / FBI Cyber / Secret Service routing
table, with concrete URLs / contacts. Currently the recipient gets
generic UK/Canada/Australia guidance and an EMPTY contact column.

### F4. Empty contact column in Suggested Filing Routes

Tied to F3: even where the Suggested Filing Routes table renders, the
"Contact" `<td>` is empty (`<td class="small"></td>`). The detective
gets no IC3 URL, no FBI portal link. The whole section is non-actionable.

### F5. Asset description is template-placeholder gibberish

Section 4 "Stolen Asset Details":
```
Asset symbol:        USDT
Token contract:      0xdAC17F958D2ee523a2206206994597C13D831ec7
Issuer:              Tether
Asset description:   ERC-20 yield-bearing wrapper token   ← WRONG
```

USDT is a 1:1 USD-backed stablecoin, NOT a "yield-bearing wrapper
token." This factual error in the headline asset description would
immediately make a sophisticated reader (federal cyber agent, law firm
partner) discount the entire package. Description appears to be a
fallback string applied without per-token routing.

### F6. Section 5 lists 407 unlabeled wallets (out of 414)

The "Identified Wallets" table dumps every counterparty the BFS touched
— 407 rows labeled `Unlabeled (under investigation)`. A detective
reading this has no signal to act on. They need the 5–20 wallets that
actually matter (perpetrator EOAs, bridge contracts, exchange deposit
addresses) with role tags ("Stargate Bridge router", "Binance hot
wallet"), not 400+ raw hex strings.

Either the label DB isn't being consulted at render time, or the
filtering threshold (which counterparties make it into Section 5) is
too permissive. Probably both.

The zero-address `0x0000…0000` is listed as "Unlabeled (under
investigation)" — should be hard-coded as "Burn / null address".

### F7. Operator name unconfigured — attestation block is unsigned

Cover meta + Section 9 Investigator Attestation both render
`(operator name not configured)`. The attestation section is the
single most legally-significant element of the package — it's the
human accountability hook. An unsigned attestation block tells the
recipient "this was generated by software and nobody is standing
behind it." Production usage requires this env var set + a CI gate
that refuses to ship a brief without it.

## Tier-2 (HIGH — would erode credibility but not catastrophic)

### F8. Asset Issuer label is ambiguous on cover

Cover meta has two adjacent fields:
- "Stolen Value (Initial)": USD 21,317.94 USDT on Ethereum
- "Asset Issuer": Circle

A reader assumes "Asset Issuer" refers to the stolen asset (USDT) → so
Circle is wrong (Tether issues USDT). The actual intent is "Freeze
Target Issuer" (recovery proceeds via Circle because the perpetrator
converted USDT → USDC). Label should be explicit:

  Stolen Asset Issuer:        Tether
  Recovery Target Issuer:     Circle

### F9. Executive Summary undersells the loss

Section 1 narrates only the USDT theft ("20,610.336829 USDT") but the
cover meta and timeline both show TWO theft events (a 0.21 ETH drain
THEN the USDT sweep). A detective reading the summary thinks the case
is one stablecoin transfer; reading the timeline sees two events
totaling $21,317.94. Summary should aggregate or explicitly enumerate.

### F10. Section 4 amount-total adds different units

Section 4 "Amount (total across 2 events)": `20,610.547644`. That
number is `0.210815 ETH + 20,610.336829 USDT` summed as if they were
the same unit. Mathematical nonsense — and any forensic-aware reader
would notice. Either keep the amount column as USD-only (`USD
21,317.94`) or split by token.

### F11. "7-day window" is template boilerplate

Section 3 timeline header: "This drain involved 2 unauthorized outbound
transfers from the victim wallet within a 7-day window". The two
transfers are on Jan 15 and Jan 16 — a **1-day window**, not 7. The
"7-day" appears to be either a default lookback constant or static
template text. Should reflect actual elapsed time.

### F12. Recoverable-positions table only shows FREEZABLE (4 of 7)

Section 4.1 narrative says "7 wallets hold recoverable USDC
positions". Table below shows 4 FREEZABLE rows. The 3 INVESTIGATE
addresses ($660,938.20) aren't in the table — reader has to scan
Section 5 (407-row list) to find them. Single unified table with a
Status column would solve it.

### F13. Service Providers table is incomplete

Section 7 lists only "Circle" as a service provider. The case touched
Ethereum (Etherscan, our infra dependency), at least one DEX (the
USDT→USDC swap), and presumably several bridges/aggregators. A
complete picture is "here are all the parties the funds touched and
why each matters" — not just the one we're asking to freeze.

### F14. Investigator Attestation date is blank

Signature block ends `Date: ___________________`. With the operator
unconfigured this is doubly weak. Both should auto-fill at brief
generation time (date = generation timestamp; operator = configured
name), with the operator able to optionally re-sign physically before
transmittal.

## Tier-3 (MEDIUM — polish items)

### F15. Whitespace in `<dl class="meta-grid">` markup looks generated

Lines like `<dt>Citizenship:</dt><dd>USA (Texas)</dd><dt>Residential address:</dt>...` jammed onto one line — Jinja template whitespace handling. Doesn't affect the rendered output but is a tell that the template is doing string-mashing rather than block emission. Pretty-printed output would help debuggability.

### F16. Counterparty `Contract` vs `EOA` correctness

Section 5 shows mixed `Contract` / `EOA` types — but for 407 rows, the
boolean must come from somewhere. Spot-check: the zero address is
classified as `EOA`. Worth confirming the contract-detection cache is
working at scale.

### F17. Footer brand line "Recupero · Investigation Services"

Inconsistent with cover wordmark "Recupero Investigation Services"
(no separator). Trivial — pick one.

## Tier-4 (LOW — items I would not block on)

- The CSS is excellent — banker/law-firm aesthetic is well-executed.
- The page-numbering footer is professional.
- Etherscan transaction links are correct and clickable.
- The 5.5 Live Filing Status pre-populated placeholder text is
  reasonable for a pre-transmittal state.
- The internal IDs (`BRIEF-ALEC-TES-356787`) are operationally useful.

## Recommended v0.30.0 scope

In priority order — first 5 close the credibility gap that would
otherwise sink a real LE pitch:

1. **F1 (version footer)** — 5-minute fix, big credibility win.
2. **F2 (PII leak in freeze letter)** — privacy compliance, must-fix
   before any real-customer use.
3. **F3 + F4 (jurisdiction misdetection)** — US victims are the bulk
   of the market; broken routing for them is unshippable.
4. **F6 (407 unlabeled wallets)** — the single largest credibility
   hit; the brief looks naive without label-DB enrichment surfacing.
5. **F7 (operator name CI gate)** — refuse to ship a brief without
   an attested operator. Same gate also covers F14.

After 1-5 land, re-run the read-through against ZIGHA-VERIFY to
confirm the same fixes hold for that case shape.

Findings F8–F17 close in v0.30.1 / .2 as polish.
