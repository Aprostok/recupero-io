# v0.28 design: identified-but-non-freezable artifact family

Spec only — NOT for v0.27.2. Per Jacob's v0.27.1 Zigha review (proposal
d): "doesn't have to ship in v0.27.1 but worth defining the shape
now."

## Problem statement

The current artifact family (freeze_request, le_handoff, victim_summary,
engagement_letter, recovery_snapshot, trace_report) assumes the
recovery path goes through an issuer with freeze authority. For
positions where this isn't true — DAI (permissionless), WETH (canonical
wrapper, no issuer), Sky-issued tokens (no individual freeze), USDS
(same), and most native tokens after first wrap — there is no
freeze letter to send. UNRECOVERABLE is a dead-end label.

But these positions are NOT actually dead ends in real recovery work.
The Zigha case is the canonical example: $18M+ of dormant DAI on
Ethereum sitting at three EOAs. No issuer can freeze it. But:

- The receiving address has an on-chain history that points to
  off-chain identifiers (CEX deposits, dApp logins, social-recovery
  contract counterparties, ENS domain registrations, etc.).
- Each off-chain identifier has a subpoena recipient (CEX, ISP, KYC
  provider, ENS registrar).
- A patient pivot through 2-3 subpoena rounds typically surfaces
  perpetrator identity, which then unlocks a seizure order.

The product gap: Recupero generates rich forensic data on these
positions but offers operators no document to send to anyone. The
data exists in `trace_report.html` and `investigator_findings.csv` but
those are operator-internal — not formatted for a subpoena recipient.

## Audience

| Recipient type | Why they get a subpoena |
|---|---|
| **ISPs** (Comcast, Fastweb, Vodafone, Spectrum) | Off-chain login origin for an exchange / dApp that the dormant address transacted with. ISP → subscriber identity. |
| **CEXes** (MEXC, KuCoin, Bybit, OKX) | Off-ramp / on-ramp deposit at a non-frozen-able token. CEX KYC → identity. |
| **KYC providers** (Sumsub, Onfido, Persona) | Identity verification records for a wallet's owner if that wallet has ever signed a KYC attestation. |
| **ENS / TNS / Lens** | Domain registration → email / wallet linkage. |
| **dApps with KYC** (Aave Arc, Centrifuge, Maple) | Same as CEXes — KYC-on-file identity for a wallet that interacted with their permissioned products. |
| **Cloud providers** (AWS, GCP, Cloudflare) | If a dApp the perpetrator used can identify their server origin; secondary pivot. |

Each recipient has different evidentiary requirements (FBI vs. court
order vs. MLAT vs. 314(b)) and different cooperation rates. The
subpoena-target artifact must surface this metadata.

The `depends_on` graph is the key novel piece — the subpoena-target
artifact is a **playbook**, not a flat list. Operators / FBI agents
follow the DAG: subpoena-1 produces records → trigger subpoena-2 →
records produce identity → seizure order.

## Artifact shape

Two new HTML/PDF artifacts per case (parallel to freeze_request_*.html):

### `subpoena_target_<recipient_slug>_<brief_id>.html`

One per recipient. Cover page: who this is for, what evidence we
have, what records we're asking for, why this records will move the
investigation forward.

Body sections mirror the freeze_request structure:
1. Executive summary (one paragraph the recipient's general counsel reads first)
2. Factual basis (the on-chain trace narrative, anchored on this
   recipient's address(es))
3. Records requested
4. Authority / legal instrument framing
5. Investigator certification + chain-of-custody

### `subpoena_playbook_<case_id>.html`

One per case. Renders the dependency graph: which subpoenas trigger
which next subpoenas, expected window, priority ordering, branch logic
("if MEXC response shows IP X, target Comcast; if it shows linked
account Y, target Bybit"). This is the document an FBI cyber unit
reads as their workplan.

## Behavior changes upstream

The trace pipeline gains a new pass after `match_freeze_asks`:

```python
# In emit_brief.py main flow:
subpoena_targets = _extract_subpoena_targets(
    case=case,
    freeze_asks=freeze_asks,          # for the rejected non-freezable rows
    editorial=editorial,               # for the AI's identified pivots
    label_db=label_db,                 # for cex/isp recipient identification
)
brief["SUBPOENA_TARGETS"] = subpoena_targets
```

`_extract_subpoena_targets` walks the case's destinations, picks out
positions where `freeze_capability == "no"` AND has any of:

- CEX deposit address → CEX subpoena
- Bridge handoff to off-ramp → trace + CEX subpoena
- Dormant EOA with recent on-chain interaction with a labeled dApp
  → dApp subpoena
- Native ETH / WETH / DAI / Sky position → seizure order target with
  ISP pivot if the address has any web-based on-chain interaction
  history (Etherscan login, MetaMask Connect, etc.)

The label DB needs to expand: today it covers issuers + bridges +
mixers + sanctioned addresses. The subpoena artifact needs labels for
CEX deposit-address ranges, KYC-provider sentinel addresses, dApp
contracts that maintain user identity, ISP / cloud-provider Whois
ranges.

## What stays out of v0.28

- **Recipient compliance email auto-send.** The freeze-letter path
  has the auto-send infrastructure; subpoenas should be manual delivery
  for the foreseeable future (chain-of-custody + judicial authority
  prerequisites that a worker process can't establish).
- **Dynamic playbook re-planning.** v0.28 ships a static playbook
  generated at case close. v0.29+ could add a "subpoena response
  ingest" path that re-plans the next pivot based on actual response
  content.
- **Multi-victim subpoena consolidation.** When the same MEXC
  deposit address appears across multiple Recupero cases, a
  consolidated subpoena from Recupero (representing N victims) is
  much more compelling than N separate ones. This is a multi-case
  feature; same data plumbing as case_clusters from v0.23 but a
  separate output.

## Validator invariants for v0.28

INVARIANT C: every `freeze_capability="no"` destination in the brief
above $1K USD has either (a) a `SUBPOENA_TARGETS` entry referencing
it, or (b) an explicit UNRECOVERABLE_ITEMS entry with `reason` field
explaining why no subpoena pivot exists.

INVARIANT D: every `SUBPOENA_TARGETS` entry resolves its `depends_on`
references inside the same case's target list (no dangling pointers).

INVARIANT E: subpoena_target_*.html files on disk == |SUBPOENA_TARGETS|
× 1 + 1 (the playbook file).
